## Context

`App\Services\Clients\Internal\Notification\Client\NotificationClient` is the only path from the SkinnyRx app to the internal notification microservice (`/notifications/send`, `/notifications/send-template`, `/notifications/send-bulk`, `/customers`, `/customer-io/events/track`). It is wired in `App\Providers\NotificationServiceProvider::registerNotificationClient()`:

```php
$this->app->singleton(NotificationClient::class, function ($app): NotificationClient {
    $config       = $app->make(NotificationConfig::class);
    $httpFactory  = $app->make(HttpFactory::class);
    $tokenProvider = $app->make(TokenProvider::class);

    $http = $httpFactory
        ->baseUrl($config->url)
        ->timeout($config->timeout)
        ->connectTimeout($config->connectTimeout)
        ->withToken($tokenProvider->provide())
        ->withHeaders([
            'Accept'       => 'application/json',
            'Content-Type' => 'application/json',
            'User-Agent'   => 'InternalAppClient/1.0.0',
        ]);

    return new NotificationClient($http, $app->make(LoggerInterface::class), $config, $app->make(PayloadBuilderResolver::class));
});
```

The closure runs once per process. The built `PendingRequest` is then shared across every subsequent call. Inside `NotificationClient::request()` we then call:

```php
$response = $this->http
    ->withHeaders(['X-Request-ID' => $requestId])
    ->retry(...)
    ->post($endpoint, $payload);
```

`Illuminate\Http\Client\PendingRequest::withHeaders()` is defined as:

```php
public function withHeaders(array $headers)
{
    return tap($this, function () use ($headers) {
        $this->options = array_merge_recursive($this->options, ['headers' => $headers]);
    });
}
```

`tap($this, …)` returns the **same** instance, and `array_merge_recursive` accumulates values per key. So the first call sets `headers.X-Request-ID = 'notif_a…'`, the second mutates it into `['notif_a…', 'notif_b…']`, the third into `['notif_a…', 'notif_b…', 'notif_c…']`, etc. Guzzle then serialises array-valued headers as a comma-joined string, which is exactly what the notification service is logging as the `request_id` field.

In Horizon, a worker keeps the Laravel container alive between jobs, so this state survives across unrelated notifications. The first notification per worker is correct; every subsequent one is silently rejected for "concurrent request" reasons whenever an earlier (now-comma-joined) ID happens to match an in-flight key on the notification side.

Other helpers we currently chain on the singleton — `baseUrl`, `timeout`, `connectTimeout`, `withToken`, the static `Accept`/`Content-Type`/`User-Agent` — happen to be idempotent under repeated `array_merge_recursive` because they're set once at provider boot and never re-applied per call. So `X-Request-ID` is the only header that demonstrably accumulates today, but the same shape of bug could appear for any future per-call header (Idempotency-Key, X-Trace-ID, …) for as long as the singleton `PendingRequest` is reused.

## Goals / Non-Goals

**Goals:**

- Every outbound HTTP request from `NotificationClient` carries exactly one `X-Request-ID` header value, equal to the ID generated by that call's `generateRequestId()`.
- The fix is structural, not a one-off hot-patch: it has to make per-call header accumulation impossible for any future per-call header too, not just for `X-Request-ID`.
- `NotificationClient`'s public surface (`send`, `upsertCustomer`, `trackEvent`) is unchanged. No caller in `app/Jobs/`, `app/Listeners/`, `app/Services/Notification/` is touched.
- Stay inside the repo's DI conventions (CLAUDE.md §3): `final readonly`, constructor DI with `private readonly`, contracts not facades, config primitives passed in via the provider.

**Non-Goals:**

- We are not removing or changing the `X-Request-ID` scheme, the `uniqid()`-based ID format, or the request-side logging.
- We are not adding a server-side `Idempotency-Key` header. The notification microservice already de-duplicates by request ID; the bug is that it received seven of them comma-joined.
- We are not changing retry/back-off behaviour, base URL handling, bearer token caching, or the existing static headers (`Accept`, `Content-Type`, `User-Agent`).
- We are not touching `PayloadBuilderResolver` or any of the per-endpoint payload builders.

## Decisions

### Decision 1 — Inject `Illuminate\Http\Client\Factory`, build the `PendingRequest` per call

`NotificationClient`'s constructor changes from taking a pre-built `PendingRequest` to taking the HTTP `Factory`, the existing `NotificationConfig`, and the existing `TokenProvider`. Inside `request()` we build a fresh `PendingRequest` from the factory on every call. The new `request()` shape:

```php
private function request(string $endpoint, array $payload): Response
{
    $requestId = $this->generateRequestId();

    $this->logger->info(...);

    try {
        $response = $this->httpFactory
            ->baseUrl($this->config->url)
            ->timeout($this->config->timeout)
            ->connectTimeout($this->config->connectTimeout)
            ->withToken($this->tokenProvider->provide())
            ->withHeaders([
                'Accept'       => 'application/json',
                'Content-Type' => 'application/json',
                'User-Agent'   => 'InternalAppClient/1.0.0',
                'X-Request-ID' => $requestId,
            ])
            ->retry(
                $this->config->retryAttempts,
                fn (int $attempt) => $this->config->getRetryDelayForAttempt($attempt),
                fn (Throwable $e) => $this->shouldRetry($e),
                throw: false,
            )
            ->post($endpoint, $payload);

        // … unchanged success / failure handling
    }
}
```

`Factory::__call()` delegates to `createPendingRequest()->{$method}(...)`, so calling any builder method on the factory produces a brand-new `PendingRequest` each time — no shared state is reachable across calls.

**Why over the alternatives:**

- _Alternative A — keep the singleton `PendingRequest`, call `->replaceHeaders(['X-Request-ID' => $requestId])` instead of `->withHeaders(...)`._
  Rejected. `replaceHeaders` only fixes this specific header. Any other per-call header anyone adds in the future (e.g., a trace ID, a real `Idempotency-Key`) would re-introduce exactly the same shape of bug. The class would also be carrying a stateful collaborator (`PendingRequest`) for no real reason — `Factory` is the type Laravel intends for "build a request" use.
- _Alternative B — `clone $this->http` before calling `->withHeaders(...)`._
  Rejected. `PendingRequest::__clone` does not deep-clone `options`, so any mutation of `headers.*` could still leak across clones depending on Laravel patch version. It's also a non-obvious workaround that the next reader would "fix" back into `$this->http->withHeaders(...)`.
- _Alternative C — recompute headers via `withOptions(['headers' => $headers], merge: false)`._
  Rejected. Same surface as A; fragile to future changes and requires the reader to know that `merge: false` is the load-bearing detail.

### Decision 2 — Keep `NotificationClient` registered as a `singleton`

The client itself is immutable after construction (its dependencies — `Factory`, `Logger`, `Config`, `TokenProvider`, `PayloadBuilderResolver` — are all stateless or have their own internal caching). Switching to `bind` would allocate a new instance per resolution for no benefit and would also force `PayloadBuilderResolver` to be re-resolved more often than needed. We keep `singleton`.

### Decision 3 — Build the per-call request inline inside `request()`, not in a separate helper

Only one method (`request()`) ever needs the per-call `PendingRequest`. Extracting a `private function buildPendingRequest(string $requestId): PendingRequest` would be a single-caller indirection. Inline keeps the request flow readable end-to-end, in line with CLAUDE.md §2/§5 (don't introduce abstractions a single concrete use site doesn't need).

### Decision 4 — Verify with a unit test that asserts cross-call isolation

A new `tests/Unit/Services/Clients/Internal/Notification/Client/NotificationClientTest.php`:

- Uses `Illuminate\Http\Client\Factory::fake()` and `recorded()` to capture outbound HTTP requests.
- Calls `NotificationClient::send(...)` (or `upsertCustomer(...)`) twice on the **same** client instance.
- Asserts that:
  1. Both recorded requests have an `X-Request-ID` header.
  2. The two `X-Request-ID` values are different.
  3. Neither `X-Request-ID` header value contains a comma.

This is the load-bearing regression test — it pins the exact shape of the bug we just paid for in stage logs.

The existing `tests/Feature/Jobs/Notification/SendNotificationApiTest.php` mocks `NotificationClient` directly and is unaffected.

## Risks / Trade-offs

- **Risk:** `Http\Client\Factory` is the **concrete** class, not an interface (Laravel does not ship a binding contract here). Injecting it deviates from CLAUDE.md §3's "inject contracts, not facades" preference.
  → **Mitigation:** This is the documented Laravel surface (`Illuminate\Http\Client\Factory`) and is already what `NotificationServiceProvider` resolves today (it just calls `make(HttpFactory::class)` before chaining). Other clients in this repo (`app/Services/Clients/*`) follow the same pattern. We're not adding a new style, just stopping one PendingRequest leak.

- **Risk:** Recomputing `withToken($this->tokenProvider->provide())` on every call instead of once at provider boot. If `provide()` were expensive, this would matter.
  → **Mitigation:** `TokenProvider::provide()` is cached in `Illuminate\Contracts\Cache\Repository` for an hour (`TokenProvider.php:14`). Per-call cost is one Redis hit, which is dwarfed by the outbound HTTP request itself. It also fixes a latent bug we currently have: the bearer token captured at boot lives for the entire worker process, so today's token will silently outlive its 1-hour cache TTL inside a long-running Horizon worker. Per-call resolution lets `TokenProvider` refresh on its own schedule.

- **Risk:** The new constructor signature is a breaking change for anyone constructing `NotificationClient` by hand.
  → **Mitigation:** The class is only resolved via the container, both in production (`NotificationServiceProvider`) and in tests (mocked through `$this->mock(NotificationClient::class)`). A repo-wide grep for `new NotificationClient(` returns only the provider site. We update the provider in the same change.

- **Trade-off:** We move the `Accept`/`Content-Type`/`User-Agent` headers out of the provider closure into the client. They are now repeated alongside the dynamic `X-Request-ID` rather than configured once. This is intentional — the entire point is that the `PendingRequest` should be a per-call disposable, so its full setup belongs next to the call site, not in long-lived state.
